The historical table below records candidate-prompt review notes, not executable agent benchmark scores. The current automated suite contains 90 cases across 25 MCP surfaces, including keyed `file_write`/`file_patch` batches, ordered same-file operations, positive/variant/negative routing, and JSON-RPC request/notification arrays. `evals:check` validates the fixture matrix; `bun run evals:measure -- <session.jsonl>` inventories direct and statically visible nested calls, while `bun run evals:score -- <session.jsonl>` associates one `ast-mcp-eval:<id>` marker from a user evaluation prompt or direct harness record with following direct MCP calls and their successful per-call outputs. Marked nested `exec` evidence and calls carrying multiple markers fail closed because their results cannot be isolated reliably. Batch unit tests, the live MCP registry test, and the Streamable HTTP transport test validate runtime behavior.

## Runtime evidence

The requested prior session recorded 28 `file_patch`, 17 `file_write`, and 8 `file_hash` calls across 212 ast-mcp-containing exec batches, with approximately 1,038,254 ast-mcp output characters. The follow-up session now records 56 `file_patch`, 13 `file_write`, and 51 `file_hash` calls across 254 batches, with approximately 1,001,670 ast-mcp output characters and 69 mutation calls. This latest transcript includes implementation, remediation, and validation work rather than a matched task control, so it is not evidence of a regression or token-quality score. The measurement command is reproducible, while a real effectiveness claim still requires matched tasks scored for required/forbidden routing, token counts, and wall time. The stdio probe initially returned no responses for a JSON-RPC array; the native stdio adapter and end-to-end test now cover that path. The new `file_patch` preview path is covered by AST/Aider unit tests and the schema/eval matrix.
